# Why jsx-string?

jsx-string is the library for rendering JSX to HTML strings — on the server, at build time, or anywhere JavaScript runs.

## Features

- **Server-first:** renders JSX without a browser, a DOM, or a virtual DOM
- **Zero dependencies:** nothing ships to your users that you didn't write
- **XSS-safe by default:** every value is escaped; dangerous URLs are blocked
- **Async-native:** any component can be `async`, `renderToString` awaits the whole tree
- **No reactivity overhead:** no hooks, no state, no reconciliation — just functions
- **Fully typed:** JSX and props carry TypeScript types end to end

## Design principles

Here are five principles that explain why jsx-string exists and what it optimizes for.

jsx-string is…

1. **[Server-first](#server-first):** your JSX runs on the server, not in the browser.
2. **[Safe by default](#safe-by-default):** security is built in, not bolted on.
3. **[No overhead](#no-overhead):** nothing between your JSX and a string of HTML.
4. **[Async-native](#async-native):** data fetching is a first-class feature, not an afterthought.
5. **[Familiar](#familiar):** same JSX syntax you already know.

---

### Server-first

**JSX was invented for browsers. jsx-string brings it to the server.**

React, Preact, and Solid all have a story for server rendering — but it is secondary. Their architecture is built around a virtual DOM, a reconciler, and a hydration story. None of that applies when the only output is a string.

jsx-string is designed exclusively for server-side HTML generation. There is no client build, no hydration, no runtime in the browser. This lets it make different trade-offs: zero dependencies, no virtual DOM allocation, and a security model that assumes all output will be trusted HTML sent to a client.

<CodeBlock code={`import { renderToString } from "@cjean-fr/jsx-string";

const html = await renderToString(
  <html lang="en">
    <head><title>Hello</title></head>
    <body><h1>Hello, world!</h1></body>
  </html>
);`} language="tsx" />

Use it to generate: static sites, email templates, SSR responses in any framework, PDF sources, API responses that return HTML fragments — anything where the output is an HTML string.

---

### Safe by default

**Security is not a configuration option. It is the only mode.**

With template literals or raw string concatenation, XSS is one forgotten escape away. jsx-string makes the safe path the only path:

<div class="grid grid-cols-1 gap-4 sm:grid-cols-2 mb-6">
  <div class="rounded-xl border border-red-200 dark:border-red-900 p-5">
    <p class="text-xs font-semibold text-red-600 dark:text-red-400 mb-3">Without jsx-string</p>
    <CodeBlock code={`// 🚨 XSS if name contains HTML
const html = \`<h1>Hello, \${name}!</h1>\`;`} language="ts" />
  </div>
  <div class="rounded-xl border border-green-200 dark:border-green-900 p-5">
    <p class="text-xs font-semibold text-green-600 dark:text-green-400 mb-3">With jsx-string</p>
    <CodeBlock code={`// ✅ name is always escaped
const html = await renderToString(
  <h1>Hello, {name}!</h1>
);`} language="tsx" />
  </div>
</div>

The security model covers three surfaces automatically:

- **Text content** — every `{value}` embedded in JSX is HTML-escaped
- **Attribute values** — `href`, `src`, `data-*` — all escaped
- **Dangerous URLs** — `javascript:` and `vbscript:` schemes are blocked in `href` and `src`

The only escape hatch is `raw()`, which is explicit and opt-in. See [Security](/guide/security) for the full security model.

---

### No overhead

**Zero runtime dependencies. No virtual DOM. No reconciliation.**

Most JSX runtimes allocate virtual DOM nodes, build a tree, diff it against a previous tree, and only then produce HTML. jsx-string skips every one of those steps. It walks the JSX tree once and concatenates strings.

The result is a library with zero npm dependencies and a gzip budget under 2 kB. There is nothing to install beyond the package itself, nothing that upgrades unexpectedly, and nothing that gets tree-shaken incorrectly.

This also means the mental model is simple: a component is a function, `renderToString` calls it, and you get a string. No lifecycle, no effect, no re-render.

---

### Async-native

**Any component can be `async`. `renderToString` awaits the whole tree.**

Data fetching in most server rendering stories is awkward. You fetch at the route level, pass everything down via props, and hope no nested component needs something you forgot. jsx-string inverts this: the component that needs data is the component that fetches it.

<CodeExample src="guide/quick-start/async-component.tsx" />

`renderToString` traverses the tree depth-first and awaits every async component it encounters. Sibling subtrees are evaluated concurrently. The result is a single `Promise<string>` that resolves when the full document is ready.

For cases where you want the HTML shell immediately and deferred content to arrive later, see [jsx-flow](/jsx-flow/overview) — an optional companion that adds streaming and static fragment generation.

---

### Familiar

**The JSX you already know. No new language to learn.**

jsx-string uses standard JSX — the same transform that React, Preact, and Solid use. You configure it the same way:

<CodeBlock code={`// tsconfig.json
{
  "compilerOptions": {
    "jsx": "react-jsx",
    "jsxImportSource": "@cjean-fr/jsx-string"
  }
}`} language="json" />

Components are plain functions. Props are plain TypeScript interfaces. Children are passed the same way. If you have written React components before, you already know how to write jsx-string components.

The main difference: there is no state, no effects, no hooks. On the server, you do not need them. Remove that complexity and what is left is just functions returning JSX — which is exactly what jsx-string renders.
